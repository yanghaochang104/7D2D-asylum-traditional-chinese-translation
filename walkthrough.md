# 7 Days To Die 模組全面繁體中文化總結報告

您好，本次的浩瀚翻譯工程（89 個模組、1.6萬行文本）已成功啟動並完成建構。這份報告總結了我們為這個專案所做的所有決策與技術實作。

## 🎯 達成目標

我們不僅成功「翻譯」了所有模組，還透過更進階的技術確保了「翻譯品質」與「維護便利性」。
原本散落在 89 個資料夾中的 `Localization.txt` 檔案，以不破壞任何原版內容為最高指導原則，完美的整合進了單一的語言包模組。

## 🛠️ 技術實作與亮點

### 1. 外掛覆蓋式管理 (Absolute Last Loading)
> [!TIP]
> **無損修改原則**：我們完全沒有更動到原本 89 個模組的檔案。

我們建立了一個名為 **`zzzzzzzzzz_TraditionalChineseTranslation`** 的全新 Mod。
利用遊戲以資料夾字母排順序讀取模組的機制（10 個 z 確保它在任何常規模組之後），讓這支模組成為遊戲中最後載入的語言包，完美且乾淨地覆蓋前面所有英文文本。

### 2. 原廠官方術語動態字典 (Dynamic Glossary Generation)
> [!IMPORTANT]
> **原版體驗對查**：沒有甚麼比看到「深度刀傷」取代官方「深切口」更破壞沉浸感。

我們捨棄了硬編碼對照表，直接透過 Python 腳本將遊戲本體 `Data/Config/Localization.txt` 吃進記憶體，解析出超過萬筆繁中官方資料作為「絕對真理字典」。

### 3. 三重智慧翻譯管線 (Smart Translation Pipeline)
腳本以秒速在背景過濾完 16,081 行文本，其邏輯如下：
* **第一重 (簡轉繁與強校正)**：掃描內建簡體的 Mod 時，將其抽取出並直接套用「原版真理字典」。若簡體寫作「深度刀伤」，即便作者這樣寫，腳本也會因為發現英文對應是「Deep Cuts」而將其扭轉為官方繁中的**「深切口」**。
* **第二重 (標籤防護)**：面對純英文 Mod，腳本會透過正則表達式洗出遊戲色碼 `[FF0000]` 並替換成安全代號 `<0>` 後發送機翻，防止代碼跑版。
* **第三重 (免授權背景機翻)**：把英文連線至 Google 翻譯引擎，取回後還原色碼並做最後一波術語覆蓋。

## ⚖️ 面對萬行文本的妥協：外部機翻與自定義擴充

> [!WARNING]
> **外包機翻的雙刃劍**：雖然這支 Python 腳本強大，但我們依然有意識地做出了取捨。

面對這龐大的 1.6 萬詞，全靠人力或 LLM 結合上下文來翻譯會面臨耗時過長的問題，因此我們將**「非原版獨創單字」**的苦力活外包給了 Google Translate。

### 潛在的語境偏差
如果 Mod 作者自創了一把名為「*Deranged sorcerer*」的新武器或史詩裝備，因為這個字「**不存在於原版官方字典中**」，攔截器會直接放行交給 Google。這就可能導致它被翻成枯燥無味的「*錯亂的男巫*」，而失去我們原本討論時所期望的「*癲狂巫師*」這種魂系帶入感。

### 如何自行維護與精進您的宇宙？
針對這個我們已經預料到的特例，我們替您在 `TranslationEngine.py` 腳本的第 41 行保留了**「手動強校正術語字典」**區塊：

```python
# 2. 加入手動強校正術語字典 (針對 Mod 翻譯容易翻車的特定名稱)
sc_to_tc['Deranged sorcerer'] = '癲狂巫師' 
sc_to_tc['Frostbite Catalyst'] = '凜冬冰晶'
```

**自訂詞庫流程：**
1. 當您在遊戲中發現某把武器被翻得太難聽，請記下它的英文原文。
2. 用筆記本打開 `Mods\zzzzzzzzzz_TraditionalChineseTranslation\TranslationEngine.py`。
3. 把您的神來一筆寫進前方的字典中（如上圖範例）。
4. 在該資料夾點兩下再次執行 Python 腳本。
5. 3 分鐘後，您的自定義霸氣台詞就會徹底壓過 Google 的冷硬機翻，成為您《7 Days To Die》宇宙的新定律！

這套「先求有（機翻打底），保核心（官方字典），再求精（手動客製）」的策略，保證了您未來的漢化體驗無懈可擊。
